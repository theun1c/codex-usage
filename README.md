# Шпаргалка по использованию Codex-агента

Этот репозиторий — практическая шпаргалка по запуску, настройке и применению Codex-агента в реальных проектах: от первого старта до работы со **skills** и **MCP**.

---

## 1) Что такое Codex-агент

**Codex-агент** — это ИИ-ассистент для инженерных задач, который умеет:

- анализировать кодовую базу;
- вносить изменения в файлы;
- запускать команды, тесты и проверки;
- делать коммиты и готовить Pull Request;
- работать по локальным инструкциям проекта (например, через `AGENTS.md`);
- расширяться через Skills и MCP-интеграции.

По сути, это «оператор разработки», который действует внутри вашего репозитория и терминала.

---

## 2) Базовый сценарий установки и первого запуска

> Конкретные команды установки могут отличаться в зависимости от вашей ОС и способа доступа (CLI, IDE, облачное окружение).

### Шаг 1. Подготовить окружение

Обычно нужны:

- Git;
- доступ к API/аккаунту;
- терминал в директории проекта.

Если вы используете **Arch Linux**, минимальный набор можно поставить так:

```bash
sudo pacman -Syu --needed base-devel git nodejs npm python python-pip
```

### Шаг 2. Настроить доступ

Как правило, нужно:

- авторизоваться в инструменте Codex;
- задать необходимые переменные окружения или профиль;
- проверить, что агент видит текущий репозиторий.

### Шаг 3. Создать конфиг

Чаще всего используется файл вида `codex/config.toml` (или эквивалент в вашей среде).

Пример структуры (шаблон):

```toml
# codex/config.toml

[model]
name = "gpt-5.2-codex"
reasoning_effort = "medium"

[project]
root = "."

[sandbox]
mode = "workspace-write"   # пример: read-only / workspace-write / danger-full-access
network_access = true

[tools]
shell = true
git = true

[mcp]
enabled = true
# servers = ["filesystem", "github", "..."]
```

Готовые варианты профилей смотрите в `docs/config-templates/codex.config.templates.toml`.

## 3) Что такое `AGENTS.md` и зачем он нужен

`AGENTS.md` — это файл инструкций для агента внутри репозитория.

Что обычно туда кладут:

- кодстайл и архитектурные правила;
- как запускать тесты и линтеры;
- правила по коммитам и PR;
- ограничения по инструментам;
- требования к финальному отчёту.

### Важный принцип области действия

- Инструкции из `AGENTS.md` действуют на директорию, где он лежит, и её поддиректории.
- Если есть несколько `AGENTS.md`, более «глубокий» имеет приоритет для своих вложенных папок.

Это позволяет задавать разные правила для разных частей монорепозитория.

Примеры:
- `examples/agents/AGENTS.python.md`
- `examples/agents/AGENTS.node.md`

## 4) Skills: что это и как использовать

**Skill** — это переиспользуемый пакет инструкций/процессов для типовых задач.

Примеры задач для skills:

- создание нового сервиса по шаблону;
- настройка CI;
- миграции БД;
- подготовка релизных заметок;
- установка и подключение внешних возможностей.

Обычно skill включает:

- `SKILL.md` с пошаговым workflow;
- дополнительные `references/` (справочные материалы);
- `scripts/` для автоматизации;
- `assets/` с шаблонами.

Практические кейсы и пример с `obra/superpowers`: `docs/skills/practical-cases.md`.

## 5) MCP: что это такое

**MCP (Model Context Protocol)** — протокол, через который агент подключается к внешним источникам данных и инструментам.

Простыми словами: MCP даёт агенту «каналы» к нужным системам, чтобы он работал не только с локальными файлами, но и с внешним контекстом.

### Что может давать MCP

- доступ к документации и внутренним knowledge-базам;
- доступ к issue tracker / задачам;
- доступ к API и сервисам компании;
- доступ к БД/схемам (в безопасном режиме);
- специальные действия через MCP-серверы.

### MCP Market

**MCP Market** обычно понимают как каталог/витрину MCP-серверов и интеграций, где можно:

- посмотреть доступные коннекторы;
- выбрать нужный сервер по задаче;
- подключить его в конфиге и проверить права доступа.

> Перед подключением MCP-серверов важно проверить безопасность: какие данные читаются, какие действия разрешены, какие токены используются.

Примеры конфигураций: `docs/mcp/mcp-server-examples.md`.

## 6) Как добавить Skills и MCP в проект

### Skills

1. Определите, какие повторяющиеся задачи есть в команде.
2. Оформите их как skill (инструкции + шаблоны + скрипты).
3. Храните skills в едином месте и версионируйте.
4. Добавьте примеры вызова в README/внутренний handbook.

### MCP

1. Выберите MCP-сервер под задачу (документация, Git-платформа, трекер и т.д.).
2. Добавьте его в конфиг.
3. Настройте переменные окружения/секреты.
4. Проверьте, что агент видит ресурсы MCP.
5. Зафиксируйте правила доступа и аудит действий.

## 7) Интеграции: где можно работать с агентом

В зависимости от вашей инфраструктуры Codex-агент обычно применяют:

- в **терминале/CLI** (локально или в удалённом dev-контейнере);
- в **IDE** (через плагины/расширения или встроенные ассистенты);
- в **веб-интерфейсах** для агентных задач;
- в **CI/CD** сценариях (автопроверки, генерация изменений, рутинные PR).

Практически полезный подход: запускать агента из того же окружения, где выполняются ваши тесты и сборка.

---

## 8) Рекомендуемый рабочий процесс

1. Описать задачу (контекст, цель, ограничения).
2. Дать агенту доступ к проектным инструкциям (`AGENTS.md`).
3. Попросить план изменений.
4. Выполнить изменения и прогнать тесты.
5. Получить дифф и объяснение, почему так сделано.
6. Сделать коммит и PR.
7. Провести human-review перед merge.

---

## 9) Полезный шаблон запроса к агенту

```text
Контекст: <что за проект и модуль>
Цель: <какой результат нужен>
Ограничения: <кодстайл, версии, нельзя менять X>
Проверки: <какие тесты/линтеры запустить>
Формат ответа: <кратко/подробно, нужен ли diff/PR>
```

Чем точнее задача и ограничения, тем стабильнее и предсказуемее результат.

---

## 10) Best practices

- Держите `AGENTS.md` коротким и конкретным.
- Не давайте агенту лишние права (принцип минимальных привилегий).
- Обязательно запускайте тесты перед коммитом.
- Используйте skills для повторяемых задач.
- Подключайте MCP только там, где он реально даёт ценность.
- Любые изменения агента должны проходить ревью человеком.

## 12) Короткий итог

Если описать в двух фразах:

- `AGENTS.md` задаёт **правила работы** агента в проекте.
- Skills дают **переиспользуемые сценарии**.
- MCP даёт **подключение внешнего контекста и инструментов**.

Вместе это превращает Codex-агента из «чат-бота» в полноценный инженерный инструмент для повседневной разработки.
